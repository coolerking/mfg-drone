# 安全要件とガードレール

**このプロジェクトで最も重要なドキュメント。** 機能追加・変更の前に必ず読むこと。
ここに記された安全機構は、テストを通すためや実装を簡単にするために弱体化・無効化しては
ならない。変更が必要な場合は、確定前に必ず人間のレビューを経ること（AI エージェント単独で
確定しない）。

## 基本原則

1. **人間が起点。** 離陸（`takeoff`）は必ず人間の明示操作を起点とする。ソフトウェアが
   自律的に離陸することは禁止。
2. **どの状態からでも止められる。** 緊急着陸は他のいかなる処理よりも優先し、状態ガードを
   無視して常に受理する。
3. **異常時は安全側に倒す（fail-safe）。** 通信断・映像断・例外発生時は、まずホバリング、
   続いて着陸へ向かう。判断に迷う実装は「止める / 降ろす」を選ぶ。
4. **モックで検証してから実機。** ロジックと UI はモックモードで検証する。実機での飛行確認は
   人間が立ち会う独立したステップとする。

## 必須の安全機構

### S-1 緊急着陸（最優先・常時受理）

- UI 上の「緊急着陸」ボタンと割り当てキー（例: スペース）は、飛行中常に有効。
- 押下時は他の処理に割り込んで着陸コマンドを発行し、状態を `LANDING` にする。
- 追従ループや手動コマンドより優先度を高くする（キューの先頭 / 専用経路）。

### S-2 コマンド送信レート制限

- 機体へのコマンド送信に上限レートを設ける（特に追従ループの `rc` 連投）。
- バックプレッシャ時は最新の指令のみ採用し、古い指令は捨てる（コマンド渋滞の防止）。

### S-3 速度・移動量の上限（クランプ）

- 手動・自動を問わず、各軸の速度 / 移動量に上限を設定し、必ずクランプする。
- 追従の制御出力も上限内に収め、急激な加減速・旋回を抑制する。

### S-4 ウォッチドッグ（通信・映像・UI 断の検知）

- WebSocket の heartbeat 途絶（ブラウザクラッシュ / タブ閉じ / ネットワーク断）を検知。
- 映像ストリームの途絶を検知。
- いずれかを検知したら、追従を停止 → ホバリング → 一定時間内に復帰しなければ自動着陸。
- Tello 本来の「コマンド無通信 約15秒で自動着陸」に頼り切らず、独自に短い猶予で安全側へ。

### S-5 対象ロスト時のフォールバック

- 追従中にトラッカーが失敗 / 対象が画面外になったら、即座に `rc 0 0 0 0`（ホバリング）。
- 状態を `TARGET_SELECTED` に戻し、UI に通知する。自動で探索旋回などはしない（初版）。

### S-6 電池残量による保護

- 電池残量が閾値を下回ったら UI に警告。さらに低い閾値で離陸を禁止 / 着陸を促す。

### S-7 飛行中の切断禁止

- 機体が飛行中の `disconnect` は受理しない。先に着陸させる（F-07）。

### S-8 状態遷移の強制

- 通常コマンドは状態機械が許可した遷移のみ発行する。状態を無視した直接送信を禁止
  （唯一の例外が S-1 の緊急着陸）。

## 実機テスト時の運用ルール（人間向け）

- 初飛行・追従の初検証は、広い屋内・天井高に余裕のある場所で、プロペラガードを装着して行う。
- 緊急着陸ボタン / キーの位置を、飛ばす前に必ず確認する。
- 人や顔・通行のある場所を対象にしない。物（箱・マーカー等）で検証する。
- 航空法・施設ルール・第三者の安全に配慮する。Tello は 80g の模型航空機扱いだが、
  規制対象外＝何でも可ではない。

## Claude Code への指示（再掲）

- S-1〜S-8 に関わるコードを変更・削除・バイパスする提案をする場合は、その旨を明示し、
  人間の承認を得てから確定すること。
- 「テストを通すため」「実装を単純にするため」に安全機構を緩める変更は禁止。
- 安全機構自体のテスト（`tests/test_safety.py`）を削除・スキップしないこと。
